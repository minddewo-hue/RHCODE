# Mobile Connection

RHZYCODE Desktop is the authoritative control service for Mobile. A mobile client needs only the desktop's LAN IP address, control port, and persistent access key.

## Desktop connection values

Open Desktop Settings and find **Mobile connection**. The panel shows:

- **IP address**: the preferred physical LAN IPv4 address.
- **Port**: `8790` by default; edit and save it directly in this panel when another port is required.
- **Access key**: a persistent `rhzy_...` credential generated by Desktop.

Desktop listens on `0.0.0.0:8790` by default so both a physical phone on the same LAN and an ADB reverse connection can reach it. Saving a new port restarts the control listener, persists the selection in `desktop-settings.json`, and disconnects existing mobile sessions so they can reconnect on the new port. Update the saved computer on Mobile to use the same port. `RHZYCODE_SYNC_PORT` remains the initial default when no saved port exists; `RHZYCODE_SYNC_HOST` still overrides the listener address for development.

The access key is generated once, survives desktop restarts, and remains valid until the user selects **Regenerate key**. It is stored only inside the Windows DPAPI-encrypted `mobile-access-state.bin` file. Provider credentials, Codex authentication, and the mobile access key must never be logged or committed.

## Authentication contract

Use the access key directly for HTTP requests:

```http
Authorization: Bearer <desktop-access-key>
```

Use these WebSocket subprotocols for `/v1/events?after=N`:

```text
rhzycode.v1
rhzycode.auth.<desktop-access-key>
```

The persistent desktop key can read state, resolve approvals, and control tasks. Remote write commands still require endpoint-specific validation and the `Idempotency-Key` rules defined by the control plane.

Regenerating the key invalidates the previous key immediately. Existing WebSocket sessions authenticated with the previous key close with application code `4001`; subsequent HTTP requests and reconnects return `401`.

## Network boundary

The LAN connection uses HTTP/WS because a physical phone cannot normally trust a certificate issued directly to a local IP address. Use it only on a trusted private network and do not expose port `8790` through router port forwarding or a public firewall rule. The access key authenticates requests but does not encrypt plaintext LAN traffic.
